## Software Architecture Report: Backstage

**Tooling Declaration**
The C4 diagrams in this report were generated using PlantUML, leveraging the standard `C4-PlantUML` library.

### 1. Context Level (Level 1)

**Explanations:**
The System Context diagram gives us the "big picture" of how the software fits into the world. By mapping out the main software system alongside its users and external dependencies, this mandatory entry-point helps answer exactly who is using the system and who has an interest in building it. At the core of our architecture is Backstage, an open developer portal framework that provides a unified interface for software cataloging, documentation, and project scaffolding.

We identified three key groups of people interacting with the portal:
*Developers, who actually build and own services, browse the catalog, and read TechDocs.

* Platform Engineers, who are responsible for configuring the Backstage instance, installing plugins, and managing integrations.
* Engineering Managers, who use the portal to monitor service ownership, team structures, and overall system health.

To function as a centralized hub, Backstage needs to communicate with several external systems:

* It connects to Source Control (like GitHub or GitLab) via REST API over HTTPS to read `catalog-info.yaml` files and create new repositories.
* It integrates with CI/CD Systems (such as Jenkins or GitHub Actions) to fetch build and pipeline statuses.
* Additionally, it queries Cloud Infrastructure (like Kubernetes, AWS, or GCP) where the actual workloads run to check cluster states.
* Finally, it relies on an Identity Provider (like Okta or GitHub OAuth) to securely authenticate users and resolve groups.

### 2. Container Level (Level 2)

**Explanations**
Moving a step deeper, the Container diagram zooms inside the software system to show its major, independently deployable parts.

The system relies heavily on a Single-Page Application (SPA) frontend built with React and TypeScript. This frontend runs directly in the user's browser and hosts all the frontend plugins, including the Catalog UI, Scaffolder UI, and TechDocs reader. It communicates via JSON over HTTPS to the App Backend , a server-side runtime built on Node.js, Express, and TypeScript. This backend hosts all of the core plugins as isolated modules, with each plugin owning its own REST route prefix.

For data management, the backend connects to a PostgreSQL Database via SQL over TCP for persistent storage.To keep data cleanly isolated, each backend plugin owns a separate logical database (for example, separate databases for the catalog, scaffolder, and auth plugins). We also mapped a Redis Cache Store, which acts as a distributed cache to reduce lookup latency, again using logically isolated namespaces for each plugin. One key architectural decision we made in this diagram was treating Object Storage (such as AWS S3 or GCS used for storing pre-built TechDocs HTML assets) as an external system. This is because it is a third-party managed service that isn't actually deployed as part of the core Backstage application.

**Relationship with the Clean Architecture Blueprint**
When reviewing this container structure, we found clear relationships with the Clean Architecture blueprint, which visualizes a system as a series of concentric circles where source code dependencies must only point inward toward higher-level policies.

The outermost layer of Clean Architecture, Frameworks and Drivers, is where the concrete "details" live, such as the specific database and web frameworks. In our system, the PostgreSQL database, Redis cache, and external React frontend map directly to this outer layer. The next layer inward, Interface Adapters, is designed to shape and translate data to make it convenient for the higher levels or external artifacts. In Backstage, the Express REST APIs and the React UI components act as these adapters, bridging the gap between the user's browser or database queries and the core logic.

Finally, the inner layers—Use Cases and Entities, which house the specific software behaviors and critical business rules—are represented by Backstage's isolated backend plugins. Because Backstage forces plugins to maintain strict logical isolation down to the database level, it heavily enforces the decoupling of policies from lower-level details. This ensures that the core business rules remain independent from the specific database or web mechanisms used.
